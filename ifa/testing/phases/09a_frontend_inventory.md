# Phase 09 — Phase A.1: `ifa/frontend/` inventory

Catalog of what's currently in `ifa/frontend/`, with size, role,
and dependencies. Input to the Phase B planning step.

## Key finding

**`ifa/frontend/` is entirely V-frontend code.** None of it is
shared with pyc. Pyc has its own:
- `python.g` / `python.g.d_parser.cc` at the project root (a
  *different* file from `ifa/frontend/python.g`).
- `python_parse.cc`, `python_ifa_*` for lowering.
- `python_ifa_main.cc` for the equivalent of `compile_one_file`.

The `ifa/frontend/python.g` here is a vestigial V-era Python
grammar used by `compile_one_file` when invoked on `.py` files —
not by pyc.

### Generated vs hand-written

- **Generated** (regenerated by `make_dparser`): `v.g.d_parser.cc`
  (28,821), `python.g.d_parser.cc` (11,889) → 40,710 LOC.
- **Hand-written**: everything else → ~3,400 LOC. Of that,
  `ast_to_if1.cc` is 1,876 LOC = 55% of the hand-written V
  frontend.

### Deletion grouping (for sequencing Phase 09 Step 5)

Three groups, each independently deletable:

1. **Grammar+parser group**: `v.g`, `v.g.d_parser.cc`, `c.g`,
   `python.g`, `python.g.d_parser.cc`. Delete first — no other
   IFA code depends on these directly except through `parse.cc`'s
   dispatch table.
2. **Parser-driver group**: `parse.cc`, `parse.h`,
   `parse_structs.h`, `make_ast.cc`, `make_ast.h`, `ast_kinds.h`.
   Delete after group 1.
3. **Lowering+scope group**: `ast_to_if1.{cc,h}`, `scope.{cc,h}`.
   This is the reference material; delete LAST, after the IR
   generator has demonstrated equivalent capability coverage.

### Build system surface

- `ifa/Makefile:120`: `LIB_SRCS` doesn't include `frontend/*`; the
  frontend is in `IFA_DEPEND_SRCS` (the binary, not the library).
  So `libifa_gc.a` is *already* frontend-free. Pyc links against
  `libifa_gc.a` and doesn't pull in any of the V frontend.
- `ifa/Makefile:204-208`: regeneration rules for both grammars.
- `ifa/main.cc`: calls `compile_one_file` (the V entry point).

This means deletion is well-bounded: removing `frontend/*` mostly
affects the `ifa` binary, not `libifa_gc.a` that pyc uses.

## What `ast_to_if1.cc` exercises (preview for Phase A.2)

API surface called by V's lowering — captured by greppping the
file for `if1_*` and other constructors:

```
if1_add_send_arg          if1_label
if1_add_send_result       if1_loop
if1_alloc_label           if1_make_symbol
if1_cannonicalize_string  if1_move
if1_closure               if1_register_sym
if1_const                 if1_send
if1_gen                   if1_set_builtin
if1_get_builtin           if1_set_primitive_types
if1_goto                  new Label
if1_if                    new Sym
```

That's the candidate API surface for the IR generator. Notable:
- `if1_if`, `if1_loop`, `if1_goto`, `if1_label` for control flow.
- `if1_send`, `if1_add_send_arg`, `if1_add_send_result` for calls.
- `if1_closure` for fun definitions.
- `if1_make_symbol`, `if1_get_builtin`, `if1_const` for syms /
  constants.
- `if1_register_sym` for namespace registration.

`new Sym`, `new Label` appear too — direct construction without
going through builders. Phase A.2 will tell us when those are used
vs the builder helpers.

## Top-level entry point flow

```
main.cc:main()
  → parse.cc:compile_one_file(fn)
      → ifa_init(new PCallbacks)
      → load_one(fn)
          → parse(fn) [dispatches to v or python parser]
          → ast_to_if1(parse_ast, callbacks) [the lowering]
      → ifa_analyze(fn)  [SHARED with pyc]
      → ifa_optimize()   [SHARED with pyc]
      → ifa_cg(fn) or llvm_codegen_write_ir() [partially shared]
```

The dotted line where V-specific code ends and shared IFA library
code begins is right after `ast_to_if1` returns. Everything
upstream is in `ifa/frontend/`; everything downstream is in
`ifa/{if1,analysis,optimize,codegen}/`.

This confirms the IR generator's interface: it replaces the
`load_one → parse → ast_to_if1` chain. The generator constructs
IF1 directly via the `if1_*` APIs, then hands off to the same
`ifa_analyze → ifa_optimize → codegen` pipeline that's already
shared.

## Recommendations for Phase B

1. **`ast_to_if1.cc` is genuinely valuable reference material.**
   1,876 LOC of working code that demonstrates how to use IFA's
   IF1 construction APIs to build non-trivial programs (closures,
   patterns, control flow, type definitions, primitive operations).
   Keep it during IR generator construction; delete in
   group 3 only after equivalence.
2. **V's grammar/parser layer is pure deletion candidate.** No
   reference value for the generator (the generator skips
   parsing entirely). Delete in group 1 first.
3. **The deletion can be staged.** Group 1 deletion (parser
   tables) buys 40k LOC reduction immediately at low risk. Group
   2 (parser driver) buys another 600 LOC. Group 3 (lowering)
   waits for generator parity.
4. **`scope.{cc,h}` is V-specific but tiny (110 LOC).** Bundle
   with group 3 since `ast_to_if1.cc` uses it.

Output: Phase B (the design phase) has clear constraints. The
generator API takes shape `(if1*) → if1*` — populates the global
`if1` and returns; no source file, no scope, no AST. Reference
material lives in `ast_to_if1.cc` until Phase C step 9 (group 3
deletion).
